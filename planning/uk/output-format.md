# Factorio Reverse Diagrams — Детальний опис вихідного формату

---

## 1. Головне питання: що це за річ?

Існує кілька принципово різних речей, якими може бути цей проєкт. Вони не взаємозамінні — кожна передбачає різну архітектуру, робочий процес та користувацький досвід. Розглянемо всі варіанти перед вибором.

---

### Варіант A — Набір постерів (оригінальний формат Гінгольда)

**Що це:** 4–6 великоформатних статичних зображень (A1 / 24×36 дюймів), кожне насичене інформацією, розроблене для друку та вивішування на стіну або перегляду як зображення високої роздільної здатності.

**Прецедент:** Діаграми SimCity Гінгольда — саме це: два зображення розміром з постер, що містять десятки міні-діаграм, анотацій, посилань на код та візуальних пояснень. "Одно-сторінкові дизайн-документи" Стоуна Лібранде — та ж ідея.

**Користувацький досвід:** Ви дивитесь на це. Скануєте, збільшуєте, читаєте, слідуєте за стрілками. Взаємодії немає. Щільність І Є досвідом — все видиме одночасно, а зв'язки між системами виражені через просторову близькість та візуальні з'єднання.

**Сильні сторони:**
- Самодостатній — працює вічно, без залежностей, серверів, браузерів
- Друкований — можна буквально повісити на стіну
- Цитований — фіксована версія, стабільне посилання
- Гарний — візуальний дизайн І Є продуктом
- Поширюваний — один PNG/PDF файл, публікуйте будь-де

**Слабкі сторони:**
- Не може обчислювати — немає живих розрахунків, зміни параметрів
- Проблема масштабу — Factorio має в 5–10 разів більшу складність ніж SimCity 1989; вмістити 50 діаграм на постери вимагає або величезних постерів, або дрібного тексту
- Проблема оновлення — кожен патч Factorio інвалідує числа; передрук постерів непрактичний
- Немає поступового розкриття — все однаково видиме, що може перевантажити

**Інструмент виробництва:** Adobe Illustrator, Figma або Affinity Designer. Ручний макет. Кожен постер — дизайн-проєкт, що вимагає 40–80 годин візуальної роботи.

**Вердикт:** Необхідний як архівний/мистецький артефакт, але недостатній як єдиний формат для проєкту такого масштабу.

### Варіант B — Explorable Explanation (довга стаття з вбудованими інтерактивами)

**Що це:** Одна довга веб-сторінка (або серія з 10 сторінок-розділів), написана як проза, з інтерактивними діаграмами, вбудованими inline. Читач скролить пояснювальний текст, і між абзацами зустрічає живий віджет, яким може маніпулювати.

**Прецедент:**
- Bartosz Ciechanowski (ciechanow.ski) — напр., "Internal Combustion Engine", "GPS", "Mechanical Watch" — технічно щільні теми, пояснені через вбудовані 3D/2D інтерактивні симуляції
- Nicky Case (ncase.me) — "Parable of the Polygons", "The Evolution of Trust" — ігроподібні explorable explanations
- Red Blob Games (redblobgames.com) — інтерактивні туторіали Аміта Пателя з pathfinding, гексагональних сіток, шумових функцій
- Bret Victor — "Up and Down the Ladder of Abstraction", Nile visualization

**Користувацький досвід:** Лінійне читання з опціональною взаємодією. Ви читаєте абзац про те, як працюють стрічки, потім під ним — живий симулятор стрічки, де можна розмістити предмети та спостерігати їх рух. Ви читаєте про еволюцію, потім — графік з повзунками. Проза надає контекст та наратив; віджети — інтуїцію.

**Сильні сторони:**
- Наративна структура керує розумінням — читач вчиться по порядку
- Поступове розкриття — складність вводиться поступово
- Кожен інтерактивний віджет малий та зосереджений — не інструмент загального призначення, а демонстрація ОДНІЄЇ концепції
- Працює на мобільних (скролінг природний; віджети можуть бути зручними для пальців)
- Проза несе пояснення — інтерактив доповнює, а не замінює
- Граціозна деградація — якщо JavaScript зламається, текст все одно передає значення

**Слабкі сторони:**
- Монолітний — важко посилатися на одну конкретну діаграму без скролінгу
- Кожен віджет — окрема реалізація — немає єдиного "двигуна діаграм"
- Довше виробляти, ніж суто статичний або суто додатковий формат
- Перехресні посилання між Частинами вимагають навігації між сторінками

**Інструмент виробництва:** React/Svelte + власні inline компоненти. Кожен розділ — markdown-подібний файл з вбудованими тегами компонентів.

**Приклад структури сторінки:**

```

│  FACTORIO REVERSE DIAGRAMS                  │
│  Частина II — Транспорт та логістика        │
├─────────────────────────────────────────────┤
│                                             │
│  § 2.1 Механіка стрічок                     │

│  Тайл стрічки містить дві лінії, кожна —    │
│  послідовність дискретних слотів предметів. │
│  Предмети просуваються на одну позицію      │

│                                             │
│  ┌─────────────────────────────────────┐    │
│  │  ╔═══════════════════════════════╗  │    │
│  │  ║  [ІНТЕРАКТИВНИЙ СИМУЛЯТОР     ║  │    │
│  │  ║   СТРІЧКИ]                    ║  │    │
```

**Вердикт:** Це найсильніший формат для освіти та залучення спільноти. Це те, що сам Гінгольд пропонує як майбутній напрямок, посилаючись на Bret Victor.

### Варіант C — Інтерактивний дашборд / інструмент

**Що це:** Одно-сторінковий веб-додаток з бічною навігацією, де кожна Частина — "сторінка" з одним або кількома інтерактивними панелями діаграм. Без прози — діаграми говорять самі за себе, з підказками та легендами для контексту.

**Вердикт:** Занадто утилітарний. Втрачає ідентичність "reverse diagram". Наратив І Є суттю — без нього це просто ще один калькулятор Factorio.

### Варіант D — Гібрид: Explorable книга + витягувані постери + повторно використовувані віджети

**Що це:** Поєднує A та B. Основний deliverable — веб-сайт explorable explanation (Варіант B). З того ж джерела генеруються статичні PDF-постери (Варіант A). Окремі інтерактивні віджети можуть вбудовуватись окремо (для постів Reddit, форумних вбудувань, інтеграції у вікі).

**Конкретна структура:**

```
factorio-reverse-diagrams.github.io/
│
├── /                          Головна сторінка з оглядовою діаграмою Частини X
│
├── /part/simulation-core      Частина I як explorable розділ
├── /part/transport             Частина II
├── /part/production            Частина III
├── ...                        (10 сторінок-розділів)
│
├── /diagram/belt-simulator    Окремий віджет (вбудовуваний)
├── /diagram/recipe-dag        Окремий віджет
├── /diagram/quality-grind     Окремий віджет
├── ...                        (~20 витягуваних віджетів)
│
├── /poster/the-tick           Завантаження PDF — Постер 1
├── /poster/factory-floor      Завантаження PDF — Постер 2
├── ...                        (6 PDF-постерів)
│
├── /data                      Сирі JSON файли даних (рецепти, сутності, формули)
│
└── /ua/                       Українське дзеркало всіх сторінок-розділів
    ├── /ua/part/simulation-core
    ├── /ua/part/transport
    └── ...
```

**Як частини пов'язані:**

```

┌──────────────────┐
```

**Вердикт:** Рекомендований підхід. Максимальне охоплення, кілька режимів споживання, одне джерело істини.

### 2.2 Мобільний (< 768px viewport)

Навігація у бургер-меню. Віджети на повну ширину. Контролі під візуалізацією (не поруч). Touch-friendly: перетягування, pinch-zoom де можливо.

### 2.3 Правила розмірів віджетів

| Тип віджету | Висота десктоп | Висота мобільний | Пропорції |
|------------|---------------|-----------------|-----------|
| Симуляція (стрічка, забруднення) | 500–600px | 300–400px | 16:9 або 4:3 |
| Діаграма вузлів-зв'язків (DAG, автомат станів) | 600–800px | 400–500px | гнучкі |
| Калькулятор (рецепт, пропорції) | 300–400px | auto (стеком) | форма |
| Графік (крива еволюції, сонячна) | 350–450px | 250–300px | 16:9 |
| Таймлайн (тік, цикл інсертера) | 200–300px | 150–200px | широкий/вузький |

### 2.4 Патерни взаємодії віджетів

| Патерн | Використання | Контролі |
|--------|-------------|----------|
| **Анімація + пауза** | Симулятор стрічки, цикл інсертера, дифузія забруднення | Кнопка play/pause, крок вперед, повзунок швидкості |
| **Зміна параметру** | Крива еволюції, сонячний вихід, калькулятор рецептів | Повзунки, випадаючі списки, числові поля |
| **Розмістити + спостерігати** | Макет маяків, мережа труб, мережа електроопор | Клік для розміщення на сітці, спостереження за обчисленим результатом |
| **Дослідження графу** | DAG рецептів, дерево технологій, система систем | Панорамування, зум, клік на вузол для розгортання/фокусу, пошук |
| **Покрокове проходження** | Оцінка тіку комбінатора, логіка переривань поїздів | Кнопка "наступний тік", лічильник тіків, відображення стану |
| **Порівняння** | Foundry vs. furnace, fluid 1.1 vs. 2.0 | Панелі пліч-о-пліч, тумблер |


### 3.2 Рівень даних

Віджет читає зі спільного файлу даних:

```
data/transport.json
{
  "belts": {
    "transport-belt":          { "speed": 0.03125, "items_per_second": 15,  "tier": "yellow" },
    "fast-transport-belt":     { "speed": 0.0625,  "items_per_second": 30,  "tier": "red"    },
    "express-transport-belt":  { "speed": 0.09375, "items_per_second": 45,  "tier": "blue"   },
    "turbo-transport-belt":    { "speed": 0.125,   "items_per_second": 60,  "tier": "turbo"  }
  }
}
```

Цей JSON автогенерується з Lua-прототипів `wube/factorio-data`.

### 3.3 Режим вбудовування

Той же віджет може рендеритись окремо за адресою `/diagram/belt-simulator` для вбудовування:

```html
<iframe src="https://factorio-reverse-diagrams.github.io/diagram/belt-simulator"
        width="800" height="500" frameborder="0">
</iframe>
```

Корисно для постів Reddit, форумних обговорень, вікі-сторінок.

## 4. PDF-постер — Анатомія одного

Беручи Постер 2 ("The Factory Floor" — Частини II + III) як приклад:

### 4.1 Фізичний формат

- Розмір: A1 (594 × 841 мм) або 24 × 36 дюймів
- Орієнтація: ландшафтна
- Роздільна здатність: 300 DPI (для друку) + векторні елементи де можливо
- Колір: темний фон (відповідає естетиці UI Factorio) зі світлим текстом та кольоровими елементами діаграм
- Сітка: підкладна 12-колонкова сітка для послідовності макету

### 4.2 Процес виробництва

```
Інтерактивні віджети (React/D3/Canvas)
         │
         ▼
    SVG-експорт (по віджету, статичний знімок при стандартних параметрах)
         │
         ▼
    Figma / Illustrator (ручна композиція)
         │
         │  ← Додати: типографіку, анотації, стрілки між діаграмами,
         │     візуальний поліш, колірну гармонію, естетику у стилі Factorio
         │
         ▼
    PDF-експорт (300 DPI, CMYK для друку, RGB для екрану)
         │
         ├──→ PDF для друку (для сервісів друку постерів)
         └──→ PDF для екрану (легший, RGB, вбудований на сайті)
```

Постер НЕ автогенерується. Це дизайнований артефакт. Дані надходять з того ж джерела, але макет, типографіка, візуальна ієрархія та естетика — людські рішення.

### 4.3 Постер vs. Веб: що куди

| Елемент контенту | Веб-розділ | PDF-постер |
|-----------------|-----------|------------|
| Пояснювальна проза | Повні абзаци | Короткі анотації, виноски |
| Інтерактивна симуляція | Жива, з параметрами | Статичний знімок при стандартних налаштуваннях |
| Таблиці даних | Розгортувані, повні | Скорочені, лише ключові значення |
| Формули | Inline з поясненням | У рамці, помітні |
| Перехресні посилання | Гіперпосилання на інші розділи | Стрілки до інших областей діаграм на тому ж постері |
| Посилання на джерела | Посилання на FFF, вікі | Рядок виносок внизу |
| Контролі (повзунки, списки) | Функціональні | Відсутні — замінені на "Див. інтерактив: [URL]" |
| Версія/інфо про патч | Автооновлюється з data pipeline | Фіксується на момент друку, показано у футері |

|-------------|-----------------|----------|

| Візуальний дизайн | НІ | Макети постерів — ручні |

## 6. Двомовна стратегія

### 6.1 Структура URL

```
factorio-reverse-diagrams.github.io/part/transport          ← Англійська (за замовчуванням)
factorio-reverse-diagrams.github.io/ua/part/transport       ← Українська
```

### 6.2 Що перекладається

| Елемент | Перекладається? | Метод |
|---------|----------------|-------|
| Проза розділів | ТАК | JSON-файли i18n з ключами перекладу |
| Підписи віджетів (назви осей, легенди) | ТАК | Рядки i18n, передані як props у віджети |
| Підписи контролів віджетів (кнопки, списки) | ТАК | i18n |
| Значення даних (числа, формули) | НІ | Універсальні |
| Назви предметів/сутностей | ЧАСТКОВО | Використання файлів UA-локалізації самої Factorio (доступні у даних гри) |
| Меню навігації | ТАК | i18n |
| PDF-постери | ОКРЕМЕ ВИДАННЯ | Окремий файл Figma з накладеним UA текстом |
| Посилання на джерела (FFF, вікі-посилання) | НІ | Лише англійські джерела |

### 6.3 Перемикач мови

Тумблер у заголовку: `[EN | UA]`. Перемикання зберігає позицію скролу та будь-який стан віджетів. URL оновлюється, додаючи або видаляючи префікс `/ua/`.

| Питання | Рішення | Обґрунтування |

| Основний формат? | **Веб-сайт explorable explanation** (гібрид Варіантів B/D) | Найсильніший для освіти; власний рекомендований напрямок Гінгольда |
